# 飞腾派通用 GPIO 控制系统课程设计实施计划

## 1. 项目目标

本项目从零实现一套基于 Linux 字符设备模型的通用 GPIO 控制系统，最终使用效果可与 libgpiod 的核心 GPIO 控制能力对照，但不得复制、链接或改写 libgpiod 的源码与私有设计。

系统必须同时满足以下目标：

- 通过 `/dev/gpioX_zsh` 字符设备向用户态提供稳定、版本化的自定义 UAPI。
- 支持 GPIO 申请与释放、输入输出配置、电平读写、批量操作、边沿事件、消抖、统计和 IOPAD 配置。
- 使用 HAL 隔离通用核心与不同厂商、不同控制器的实现。
- CLI 同时支持单命令、交互式 REPL 和脚本化控制，并提供机器可解析的 JSON 输出。
- 默认路径优先保证安全性和内核资源一致性；性能优化不能破坏所有权、边界检查和可恢复性。
- 当前阶段以完成通用 GPIO 控制系统为主。板载 LED20 可作为无需额外接线的真实 GPIO 冒烟测试和演示负载，但不得把 LED20 的引脚、有效电平或控制逻辑写死在通用 core；后续专用外设驱动必须复用本项目提供的接口。

## 2. 范围与非目标

### 2.1 第一版必须实现

1. 通用字符设备核心和动态控制器注册。
2. gpiolib/pinctrl 安全后端、Phytium IOPAD 后端、mock 测试后端。
3. 每文件描述符独占租约及进程退出自动清理。
4. 单线和同一控制器内的多线批量配置、读写与事务回滚。
5. 上升沿、下降沿和双边沿事件，支持 `read()`、`poll()` 和 `epoll()`。
6. 硬件消抖优先、软件消抖回退，以及有界事件队列和溢出报告。
7. IOPAD 偏置、驱动档位和命名复用状态；通过能力查询处理平台差异。
8. sysfs 只读监控及少量管理员策略配置。
9. 原创用户态 C 库和 CLI。
10. KUnit、mock、集成、并发、故障注入、模糊测试、静态分析和硬件测试。
11. 与 libgpiod 的同板、同引脚、同负载黑盒性能对照。

### 2.2 第一版明确不做

- 不实现 GUI。
- 不提供跨控制器硬件原子事务。
- 不提供共享输入租约。
- 不通过 sysfs 控制 GPIO 电平、方向、偏置或驱动能力。
- 不向用户态暴露物理地址、寄存器偏移、位掩码或任意 MMIO 访问。
- 不为老旧、未经验证的内核堆积大量条件编译。
- 不开放 IOPAD 输入/输出延时写控制；仅在 ABI 中保留可扩展空间。
- 不在此阶段实现 LED20、传感器等专用外设驱动；允许通过设备树元数据和通用 CLI 控制板载 LED20，以验证通用接口。

### 2.3 已确定的硬件测试夹具

开发板当前可使用三盏 LED 进行可视化调试：

- 板载 LED20：用于模块加载后的基础冒烟测试和单线控制验证。
- 外接 GPIO4_7 LED：用于单线、批量、并发和事件相关测试。
- 外接 GPIO1_11 LED：用于单线、批量、并发和事件相关测试。

两盏外接 LED 必须分别串联合适的限流电阻（建议 `330Ω～1kΩ`），不得依赖 GPIO 内部上拉电阻限流。首次驱动前必须从设备树、原理图和板级映射中核对控制器、line offset、IOPAD 复用、有效电平、电压兼容性及资源占用情况。

`GPIO4_7`、`GPIO1_11` 和 `LED20` 只能出现在设备树/板级元数据、测试配置、示例脚本及测试报告中，不得写死在通用 core、HAL 接口或通用 UAPI 中。

## 3. 总体架构

```text
单命令 / REPL / 脚本 / JSON
              |
        libgpioctl_zsh
              |
    自定义 ioctl + read/poll
              |
       gpioctl_core_zsh
       /        |        \
 gpiolib HAL  Phytium HAL  mock HAL
     |         |           |
 GPIO/pinctrl 受控 IOPAD   自动化测试
```

### 3.1 模块划分

- `gpioctl_core_zsh.ko`
  - 字符设备注册、UAPI 分派、租约、事务、事件、权限、统计和资源生命周期。
- `gpioctl_backend_gpiolib_zsh.ko`
  - 默认安全后端，通过内核 GPIO descriptor、IRQ、pinctrl/pinconf 接口访问硬件。
- `gpioctl_backend_phytium_zsh.ko`
  - 飞腾专用 IOPAD 能力映射，以及严格隔离的 raw-MMIO 实验路径。
- `gpioctl_mock_zsh.ko`
  - 可编程成功、失败、超时、忙、错误读回和事件注入，不用于生产部署。
- KUnit 测试模块
  - 单独编译和加载，避免测试逻辑进入生产模块。

后端通过带内部版本号和结构长度的 HAL 操作表向 core 注册。通用 core 中不得出现飞腾物理地址、PAD 名称、固定控制器数量或板级引脚表。

### 3.2 字符设备模型

- 每个已注册控制器对应一个节点：`/dev/gpio0_zsh`、`/dev/gpio1_zsh`……
- 节点数量动态生成，不在通用 core 中固定为六个。
- 每个控制器独立维护锁、租约表、事务上下文、事件队列和统计。
- 设备节点默认权限为 `0660 root:gpio`，由 udev 规则稳定设置。
- 打开的文件描述符持有模块引用；存在活动文件或租约时禁止后端注销和模块卸载。

## 4. 自定义 UAPI 设计

公共头文件为 `include/uapi/gpioctl_zsh.h`。ioctl 是 Linux 提供的系统调用机制，但请求编号、数据结构、`.unlocked_ioctl`/`.compat_ioctl` 处理、校验、权限和业务语义全部由本项目自行实现。

### 4.1 ABI 约束

- 使用固定宽度类型，如 `__u32`、`__u64`、`__s32`。
- 每个可扩展结构含 `abi_version`、`struct_size`、`flags` 和清零的 `reserved[]`。
- 所有输入先整体 `copy_from_user()` 到内核缓冲区，校验完毕后才修改状态。
- 所有输出先在内核缓冲区形成一致快照，再用 `copy_to_user()` 返回。
- 单个标量可在适当场景使用 `get_user()`/`put_user()`，但必须检查返回值；绝不直接解引用用户指针。
- 为 32/64 位结构布局添加静态断言和 compat 测试。
- 未知 flags、非零保留字段、非法长度、溢出数量和越界 line offset 必须拒绝。
- ABI 内不传字符串和指针链；字符串解析位于用户态库及 CLI。

### 4.2 ioctl 功能组

1. `GET_ABI` / `GET_CAPS`：查询 ABI、控制器、每线能力及限制。
2. `LEASE_REQUEST` / `LEASE_RELEASE`：申请或释放一组 GPIO。
3. `LINE_CONFIG`：配置输入、输出、初值、active-low、bias 和消抖。
4. `VALUE_GET` / `VALUE_SET`：单线或掩码式多线电平操作。
5. `BATCH_EXEC`：执行有上限的同控制器批量事务。
6. `EVENT_CONFIG`：启停边沿事件并配置队列和消抖策略。
7. `GET_STATS`：读取操作、错误、拒绝、冲突和事件丢失统计。
8. `IOPAD_GET_CAPS` / `IOPAD_CONFIG`：能力化的 IOPAD 查询和受限配置。

事件数据通过 `read()` 返回固定布局记录；`poll()`/`epoll()` 表示事件可读、错误或设备移除。普通 GPIO 电平控制不复用 `read()`/`write()`，避免数据流语义和控制语义混淆。

## 5. 寻址与硬件描述

### 5.1 标准标识

- 内核 ABI 的唯一稳定标识是“当前控制器设备 + line offset”。
- 禁止依赖废弃的 Linux 全局 GPIO 编号。
- 用户态允许使用 `GPIO1_11`、`header-pin-15`、`led-left`、`BA49`、`W53` 等别名，但必须先唯一解析为控制器和 offset。
- PAD 球位名、排针号和别名均为板级元数据，不进入通用 core。

CLI 提供：

- `gpioctl_zsh list`
- `gpioctl_zsh info <controller|line>`
- `gpioctl_zsh resolve <alias>`

输出应包含控制器、offset、别名、物理排针、PAD 名、占用情况、方向和能力。别名缺失、重复、越界或指向保留引脚时必须失败，不得猜测。

### 5.2 设备树

设备树或 overlay 是硬件拓扑的唯一正式来源，描述：

- 控制器 phandle 和后端 compatible。
- 允许使用和必须保留的 line。
- GPIO 别名、PAD 名和物理排针元数据。
- 可用 IOPAD 能力及命名 pinctrl 状态。
- 每线释放时的安全状态。
- Phytium 实验后端的受控 MMIO 资源。

模块参数只允许控制日志级别、测试开关等非拓扑调试项。初始化时必须严格验证设备树；配置矛盾、资源重叠或缺少安全状态时，相关输出能力不得启用。

## 6. 所有权、并发和资源生命周期

### 6.1 租约模型

- 租约绑定打开的文件描述符，并记录在 `file->private_data` 的会话上下文中。
- 第一版只支持独占申请；多线申请必须全部成功或全部失败。
- 只有该文件描述符持有的租约才可进行配置、读写和事件操作。
- `close()`、进程异常退出、初始化失败和设备移除走同一套幂等清理路径。
- 控制器、line、租约和事件对象必须有明确引用关系，避免 use-after-free。

### 6.2 锁设计

- 可睡眠的控制路径、租约和事务状态使用 mutex。
- IRQ/事件生产路径使用最小临界区的 spinlock，并采用 `spin_lock_irqsave()` 等与上下文匹配的形式。
- 不在自旋锁内执行内存分配、用户内存复制、可能睡眠的 GPIO/pinctrl 调用或长循环。
- 定义并文档化锁顺序，至少覆盖 controller → lease → event queue；不得反向加锁。
- MMIO 寄存器的读改写在适当锁保护下完成，并使用 `readl()`/`writel()` 等访问器及必要的顺序保证。

### 6.3 批量事务

批量请求设固定最大操作数，流程为：

1. 整体复制到内核。
2. 检查结构、数量、算术溢出、权限、租约、能力和操作间冲突。
3. 预分配资源并建立可回滚快照。
4. 获取控制器事务锁。
5. 按无毛刺顺序提交。
6. 读回验证。
7. 失败时逆序回滚，并明确报告原始错误和回滚结果。

仅承诺同一控制器内的软件事务一致性；若硬件没有批量寄存器，不宣称真正同时翻转。跨控制器请求由用户态拆分，明确不保证硬件原子性。

## 7. 安全状态与无毛刺配置

- 默认未知引脚的安全状态是输入、高阻、无偏置；在设备树未确认其输出安全性时拒绝输出申请。
- 已知外设可在设备树声明安全输出值和偏置。
- 输出配置顺序应优先预装载输出值，再切换方向，减少瞬态毛刺。
- 租约释放恢复声明的安全状态，而不是简单恢复申请前可能不安全的状态。
- 事务快照只用于失败回滚，不替代租约释放时的安全策略。
- IOPAD 释放时回到 default/sleep pinctrl 状态。
- 清理函数必须可重复调用，并覆盖正常关闭、异常退出、probe 失败、设备移除和模块退出。

## 8. 边沿事件设计

事件记录至少包含：

- ABI/结构版本。
- line offset。
- 上升沿或下降沿类型。
- 单调时钟时间戳。
- 每租约或每事件流递增序号。
- 溢出、设备移除等状态标志。

事件队列采用初始化时预分配的有界环形缓冲区。IRQ 路径只执行采样、过滤、入队和唤醒，不动态分配内存。队列满时采用已文档化策略丢弃并递增 drop 计数，同时向用户态暴露可检测的 overflow 状态。

消抖策略：

1. 后端支持硬件消抖时优先使用。
2. 否则使用基于单调时间的轻量软件过滤。
3. 能力查询明确实际采用的方式和时间粒度。

## 9. IOPAD 通用能力模型

飞腾手册确认 `_reg0` 支持无偏置/下拉/上拉、16 档驱动和 `func0`～`func6`，`_reg1` 支持输入输出延时。

第一版对外规则：

- 通用偏置枚举：disable、pull-up、pull-down。
- 驱动强度仅在存在可信 mA 映射时使用 mA；飞腾第一版使用明确的 `level 0..15`。
- 正常后端使用设备树命名 pinctrl 状态表示复用，不允许普通用户传裸 `funcN`、地址或掩码。
- 每线能力可查询；不支持的能力返回 `-EOPNOTSUPP`。
- IOPAD 写配置要求调用者持有对应 GPIO 租约，且具备 root 或 `CAP_SYS_RAWIO`。
- 应用过程为完整校验、加锁、写入、读回验证和失败回滚。
- 输入/输出延时先保留 ABI 扩展位，第一版不开放写入，待有明确时序需求和测量条件后实现。

## 10. 权限与攻击面控制

- 普通 `gpio` 组用户只能操作设备树 allowlist 中的 GPIO。
- mux、pull、drive 等 IOPAD 电气配置要求 root 或 `CAP_SYS_RAWIO`。
- reserved line 永远拒绝普通申请；不得通过模块参数绕过。
- 每次操作检查当前文件的租约所有权和所需凭据。
- 所有数组长度、偏移、位移、乘法、加法和用户提供的超时值均做边界及溢出检查。
- 日志对重复非法请求限流；详细计数通过统计接口提供。
- UAPI 不接受内核地址、用户回调地址、任意寄存器地址或无限长度数据。
- raw-MMIO 后端必须 `request_mem_region()`，资源忙时返回 `-EBUSY`；禁止强制映射、自动解绑活动 gpiochip 或绕过资源所有权。

raw-MMIO 实验必须使用单独 overlay 禁用被实验后端占用的安全控制器，并记录绑定和恢复步骤。无法证明隔离安全时，只允许 mock/KUnit 验证，不能在真实硬件强行执行。

## 11. sysfs 边界

每个控制器的 sysfs 监控项可包括：

- backend、ABI 版本、line 数量和能力摘要。
- active leases、成功操作数、错误数、拒绝数、冲突数。
- event count、event drop、overflow count。
- 少量经过严格校验的管理员策略，例如统计清零或日志策略。

方向、电平、偏置、驱动、复用和批量事务全部通过字符设备 UAPI 完成。debugfs 仅用于可选调试，不构成稳定接口。CLI 的 `status` 可读取 sysfs 监控信息。

## 12. 用户态库与 CLI

### 12.1 `libgpioctl_zsh`

使用 C 独立实现，封装：

- 控制器枚举和 line 信息。
- 租约申请、配置和自动释放。
- 单线/多线读写和批量事务。
- 边沿事件、poll/epoll 和超时。
- IOPAD 能力及受限配置。
- 统计、错误码和诊断文本。

库对象持有文件描述符并负责幂等清理，不调用、不链接 libgpiod。

### 12.2 CLI 模式

CLI 的三个入口共用同一命令解析和执行引擎：

1. 单命令：适合 Shell 管道与自动化。
2. `gpioctl_zsh shell`：交互式 REPL。
3. `gpioctl_zsh run <file>` 或 `gpioctl_zsh run -`：执行脚本文件或标准输入。

脚本模式必须支持：

- 无提示运行、稳定 stdout/stderr 和退出码。
- `--json`、`--timeout`、`--dry-run`、`--strict`。
- 同一控制器内的 transaction block。
- 行号和明确错误定位。
- 确定性、简单的命令语法；循环和复杂条件交给 Shell/Python 等外部工具。

## 14. 分阶段实施与质量门

### 阶段 0：需求、基线与原创性档案

产出：需求清单、术语、架构、UAPI 草案、威胁模型、依赖清单、libgpiod 黑盒基准方法。

质量门：

- 每项课程要求都映射到设计项和计划测试。
- UAPI 不包含裸地址、平台字符串或可变指针结构。
- `DEPENDENCY.md` 明确没有 libgpiod 编译或运行时依赖。

### 阶段 1：core 骨架、mock HAL 与 KUnit

产出：动态控制器注册、会话/租约状态机、HAL 接口、mock 后端和首批 KUnit。

质量门：

- probe/remove、open/close、全成全败申请、重复清理通过测试。
- mock 可注入忙、超时、部分失败和错误读回。
- 锁顺序和对象生命周期有文档及断言。

### 阶段 2：基础 GPIO 安全后端

产出：gpiolib 后端、设备树解析、方向/电平/active-low/bias、权限和安全释放。

质量门：

- 在飞腾派上验证 GPIO1_11、W53 及其他 allowlist line。
- 两进程竞争、进程崩溃、非法 line、reserved line、权限不足均符合预期。
- 加载、失败回滚、卸载无资源泄漏和异常日志。

### 阶段 3：用户态库与完整 CLI

产出：原创 C 库、单命令、REPL、脚本、stdin、JSON、dry-run 和 strict 模式。

质量门：

- 三种 CLI 模式共享执行语义。
- Shell 可稳定依据退出码和 JSON 判断结果。
- 脚本错误能定位文件、行号和底层 errno。

### 阶段 4：批量事务

产出：有界批量 UAPI、快照、提交、读回和回滚。

质量门：

- 所有验证在第一次硬件修改前完成。
- 任意中间步骤失败后，mock 状态和真实硬件状态符合回滚约定。
- 明确验证同控制器原子边界和跨控制器非原子边界。

### 阶段 5：事件与消抖

产出：IRQ 接入、事件 ring、read/poll/epoll、序号、时间戳、overflow 和消抖。

质量门：

- 上升/下降/双边沿、超时、队列满、close 与事件并发全部覆盖。
- IRQ 上下文无睡眠和动态分配。
- 用户态能可靠检测丢事件，不出现静默丢失。

### 阶段 6：IOPAD 与 sysfs

产出：IOPAD caps、偏置、16 档驱动、命名复用、权限控制和监控属性。

质量门：

- 不支持的能力统一返回 `-EOPNOTSUPP`。
- 配置失败可回滚，释放恢复安全 pinctrl 状态。
- 普通用户无法修改高权限 IOPAD 项，sysfs 无 GPIO 控制旁路。

### 阶段 7：Phytium raw-MMIO 实验后端

产出：独立 overlay、资源申请、寄存器 RMW、实验启停和恢复文档。

质量门：

- 与现有 gpiochip/pinctrl 资源冲突时安全失败。
- 无可证明的硬件隔离时不得进行真实板卡写测试。
- raw 后端测试结果与默认安全后端结果分开报告。

### 阶段 8：安全、稳定性和性能收口

产出：全套分析结果、压力测试记录、benchmark 原始数据与报告。

质量门：

- max warning 构建、Sparse、Coccinelle、Smatch 无未解释的高风险问题。
- 条件允许时完成 KASAN、UBSAN、LOCKDEP 测试。
- 至少 1 小时混合负载无崩溃、死锁、资源泄漏和无法解释的事件丢失。
- 所有安全/正确性硬门通过后才发布性能结论。

## 15. 测试矩阵

### 15.1 单元与状态机

### 15.5 硬件测试

- 输入、输出、高低电平和读回。

- GPIO4_7 与 GPIO1_11 上的两盏外接 LED 用于单灯控制、同步/批量控制、交替闪烁、事务失败回滚和安全释放的可视化验证；外接 LED 必须分别串联限流电阻。

- GPIO 释放后检查方向、电平、偏置和复用已恢复安全状态。

## 16. 性能评估

在同一飞腾派、同一内核、同一引脚、同一 CPU 亲和性和相同测试轮次下，将本项目安全后端与 libgpiod 做黑盒对照。不得把进程启动时间混入单次 GPIO 操作延迟。

测量指标：

- 单次 get/set 的平均值、P50、P95、P99。
- 不同 batch 大小下的吞吐和单操作摊销。
- 事件速率、CPU 占用、延迟分布和 drop 数。
- 1、2、4、8 线程并发扩展性。
- 租约申请/释放时间和一小时混合负载稳定性。

目标：

- 安全后端单线操作达到同条件 libgpiod 基线的约 90% 或以上。
- 通过持久文件描述符、批量 UAPI、预分配和减少用户复制，在批量负载中争取达到或超过基线。
- raw-MMIO 数据只作为实验结果，不得用于替代安全后端的正式结论。
- 若未达目标，必须公开原始数据、瓶颈和下一步优化方案，不能牺牲安全门换取数字。

## 17. 内核与平台兼容策略

- 第一优先级：飞腾派 `6.6.63-phytium-embedded-v3.2`，完成全部硬件验证。
- 通用 core 和 gpiolib 后端以 Linux 6.6 API 为基线，不修改板载内核源码。
- 使用小型兼容层验证 Linux 6.1；条件允许再验证 5.15。
- 文档区分“已验证”“预计兼容”“不支持”，不对未经测试的版本作保证。
- CI 至少执行用户态构建、UAPI 布局检查、静态分析和可运行的 KUnit/mock 测试；板卡测试由专用脚本执行。

## 18. 原创性与可审计性

- 不复制 libgpiod 源码、私有结构、测试代码或命令语法。
- 不链接 libgpiod；构建和依赖扫描必须能证明这一点。
- libgpiod 只用于公开行为理解和外部黑盒性能基准。
- UAPI、库 API、CLI、错误模型和测试全部使用本项目原创的 `_zsh` 命名空间。
- 在文档中维护“课程需求 → 设计决策 → 实现文件 → 测试用例”追踪表。
- 引用 Linux 内核公开 GPIO/pinctrl API 或课程手册时记录来源和接口依据。
- 保留 benchmark 命令、环境、原始数据和分析脚本，保证结果可复现。

## 19. 主要风险与应对

| 风险 | 应对措施 |
|---|---|
| gpiolib 导出接口不足以支持某些 pinconf | 通过 capability 降级；不绕过资源所有权，必要时限定为 Phytium 受控后端 |
| GPIO 与 pinctrl 所有权冲突 | 统一租约和锁顺序；设备树声明资源；冲突时返回 `-EBUSY` |
| 事务无法实现硬件同时翻转 | 明确软件事务边界；利用后端批量寄存器能力，否则不宣称硬件原子性 |
| IRQ 风暴导致事件丢失 | 预分配有界队列、drop/overflow 可见、消抖和压力测试 |
| close/remove/ioctl 竞争引发 UAF | 引用计数、模块所有者、状态机、同步注销和 KUnit/LOCKDEP 验证 |
| IOPAD 配置损坏外设或造成争用 | allowlist、capability、特权检查、安全状态、读回验证和回滚 |
| 为追求性能绕过安全路径 | 正确性作为硬门；raw 结果单列；优化以批处理和减少复制为主 |
| “通用”范围无限膨胀 | 以 HAL、设备树、mock 和能力查询证明通用性，不伪造第二厂商实现 |
| 被质疑照抄 libgpiod | 原创 API、依赖扫描、设计追踪表、黑盒基准和完整决策记录 |

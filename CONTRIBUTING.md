# Contributing to gpioctl-zsh

感谢你改进 `gpioctl-zsh`。本项目包含内核模块、原创 ioctl UAPI、用户态库、CLI、
设备树和板级策略；提交前请先说明变更影响的层次和验证范围。

## 开始之前

1. 先阅读 `README.md`、`docs/architecture-zsh.md` 和
   `docs/developer-guide-zsh.md`。
2. 缺陷和功能建议优先通过 GitHub Issue 讨论。
3. 安全漏洞不要提交公开 Issue，按 `SECURITY.md` 私下报告。

## 命名与兼容性

- 项目自有模块、文件、符号和测试工件继续使用 `_zsh` 后缀，避免与其他课程模块
  或内核全局符号冲突。
- UAPI 变更必须保留版本、结构长度、固定宽度类型、清零保留字段和 compat 评估。
- 通用 core 不得加入飞腾寄存器地址、固定控制器数量、PAD 名或 LED 引脚；平台
  差异应位于 HAL、设备树或板级配置。
- 生产代码不得复制、链接或运行时依赖 libgpiod。可选 benchmark 只能使用其公开
  API，并保持在默认构建和安装目标之外。

## 安全要求

- 所有用户输入必须在产生副作用前完成格式、范围、长度、枚举和策略校验。
- 不在 spinlock、中断或其他原子上下文中访问用户内存、睡眠或分配可睡眠内存。
- 资源申请、租约、映射、IRQ 和模块引用必须具有明确的逆序回滚与释放路径。
- MMIO 只能通过 platform resource 独占映射；禁止任意地址、任意 mask 或绕过
  资源所有权的后备路径。
- 新的输出线路默认不得写入，直到设备树策略、原理图、复用、电压和接线均已核对。

## 提交前验证

至少运行：

```sh
make check
git diff --check
```

涉及内核、设备树或硬件时，还应执行与变更匹配的构建、mock、KUnit、静态分析、
并发、生命周期或实板测试。报告必须区分 `PASS`、`FAIL` 和因环境不具备而
`SKIP` 的项目，不得用替代测试冒充通过。

## Pull Request

- 一个 PR 解决一个清晰问题，避免混入无关格式化或生成文件。
- 说明变更原因、用户/开发者影响、安全边界和验证命令。
- 硬件测试注明开发板、内核版本、接线、测试时长、最终电平和模块清理状态。
- 不提交密码、令牌、私钥、固定局域网 IP 或个人绝对工作路径。

提交贡献即表示你有权按仓库的 `GPL-2.0-only` 许可证提供该贡献。
